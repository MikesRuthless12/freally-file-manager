Install Copy That

  1. Choose your platform's recommended installer from the table below.
  2. Run the one-liner in a terminal, or download the installer from the Releases page.
  3. Windows only — SmartScreen warning: the first time you launch, Windows may show a "Windows protected your PC" dialog. Click More info → Run anyway. This clears after roughly 100 downloads as Microsoft builds reputation for the binary.
  4. macOS only — Gatekeeper: double-clicking the app the first time will be refused. Instead right-click → Open → Open to whitelist the bundle. If you installed via Homebrew (brew install --cask copythat), the quarantine flag is cleared automatically — no extra step needed.
  5. Linux — AppImage: after downloading, mark the file executable with chmod +x copythat-*.AppImage then run it. On SELinux-enforcing systems this step is required before the file will launch.
  6. Launch Copy That. The main window opens at 1120 × 640 by default (minimum 960 × 420).

Copy files

  1. Launch Copy That 2026.
  2. Add your source files — drag and drop one or more files or folders onto the window, or click Add files / Add folders in the header toolbar.
  3. The drop-staging dialog opens. Set:
    • Destination — choose the target folder via the native file picker.
    • Operation — select Copy (this is the default).
    • Collision policy — what to do if a file already exists at the destination: Keep both, Skip, Overwrite, Overwrite if newer, or Ask every time.
    • Order — rearrange the source list by name or size, or drag rows manually. Files are always grouped before folders.
  4. Click Start copying.
  5. Watch progress in the queue view — each file gets its own row showing a circular progress ring, live speed (MiB/s), bytes copied vs. total, and an ETA. The header strip shows aggregate speed and overall ETA across all running jobs.
  6. To pause a job, press Space with that row focused, or use the right-click context menu. Click Resume (or press Space again) to continue.
  7. When all jobs finish, the state pill shows Idle and the completed rows show the Done badge. A verified copy also confirms byte-exact integrity (see Verify a transfer).

Move files

  1. Launch Copy That 2026.
  2. Drag and drop files or folders onto the window, or use Add files / Add folders in the toolbar.
  3. In the drop-staging dialog, set Operation → Move.
  4. Choose your Destination folder and set a Collision policy.
  5. Click Start moving.
  6. Copy That copies each file to the destination first, then deletes the source only after the destination write succeeds. If a failure occurs mid-move, the source file is preserved and the row is marked Failed — no data is lost.
  7. Completed rows show the Done badge. Check the History tab to confirm all files transferred successfully.
You can also trigger a move from Windows Explorer, macOS Finder, or your Linux file manager using the right-click "Move with Copy That" entry — see Shell & right-click menu.

Securely delete files

  1. From your file manager (Windows / macOS / Linux): right-click the file or folder and choose Secure delete with Copy That.
    From inside the app: right-click any completed job row in the queue and choose Secure delete source.
  2. A confirmation dialog shows the file path(s) and the currently configured shred method.
  3. To change the shred method for this deletion, click the method name to open the picker. For the default method across all future deletions, go to Settings → Secure delete.
  4. Click Shred (you may be asked to confirm a second time if Require double confirmation is enabled in Settings).
  5. Copy That overwrites the file data according to the chosen method, then deletes it. The row shows Done when shredding is complete.
MethodPassesWhen to use
Zero1Quick disposal of non-sensitive data
Random (CSPRNG)1Fast cryptographically random overwrite
DoD 5220.22-M3Standard compliance requirement
DoD 7-pass (ECE)7Higher-assurance compliance
Schneier7Practical strong deletion
VSITR (German BSI)7European regulatory environments

NIST 800-88 Clear1Modern magnetic-media standard
SSD warning: multi-pass overwrite does not reliably sanitize flash storage due to wear-leveling and over-provisioning. Copy That detects SSDs and shows an advisory toast. For true SSD sanitization, use ATA SECURE ERASE via your drive vendor's tool, or rely on full-disk encryption with key rotation.

Organize & batch operations

  1. Add multiple files and folders to Copy That in a single drag — or click Add files / Add folders multiple times to build up a larger source list.
  2. In the drop-staging dialog, use Order to sort the source list by name or size (ascending or descending), or drag rows manually to prioritize. Files always appear before folders regardless of sort order.
  3. To skip certain files, open Settings → Filters before starting. Add include or exclude globs (e.g., **/*.log to exclude all log files, or **/*.txt to include only text files). You can also set size bounds or modification-time bounds to filter by age or file size.
  4. Choose your Collision policy in the staging dialog — Skip is useful for batch syncs where you want to add only new files; Overwrite if newer handles incremental updates.
  5. Click Start copying or Start moving.
  6. The Activity panel (lower half of the main window) shows one row per file the engine touches as it enumerates the source tree. Rows display status badges: Pending, Running, Done, Skipped, or Failed. On trees over 250,000 files the panel stops adding rows to stay responsive, but the queue progress bar continues counting all files.
  7. Use the When done selector (top-right of the Activity panel) to automatically close the app, log off, shut down, or sleep the machine when the entire queue finishes.
  8. To re-run a batch later, go to the History tab, find the job, and click Re-run — it queues the same sources and destination with one click.

Verify a transfer

  1. Open Settings → Transfer and find the Verify drop-down.
  2. Choose a verification algorithm. Recommended choices:
    • xxHash3 — fastest option, ideal for day-to-day copies.
    • BLAKE3 — cryptographically strong and still very fast; use for important data.
    • SHA-256 — maximum compatibility with external tools and TeraCopy-format sidecar files.
  3. Close Settings. The next copy job will automatically verify.
  4. During a copy, Copy That hashes the source during the read pass — no second read of the source needed. After all bytes land at the destination, a dedicated post-pass hashes the destination file.
  5. If the hashes match, the row badge shows Done. If they differ, the row shows Failed with a VerifyFailed error — click the row to open the details drawer with the expected vs. actual hash values.
  6. To export a hash sidecar (TeraCopy-compatible format), click the export icon in the queue header or right-click a completed row and choose Export checksums. Each line is <hex> <relative path>.
If a copy appears to stall at 100%, check the state pill in the header. If it reads Verifying, the post-copy hash pass is still running — this is normal for large files or slower algorithms like SHA-256.

Shell integration & right-click menu

  1. Shell integration is installed automatically alongside the app on all platforms — no separate step is needed.
  2. Windows: in File Explorer, right-click any file or folder. You will see Copy with Copy That and Move with Copy That in the context menu. These entries are provided by an IExplorerCommand shell extension registered during install.
  3. macOS: in Finder, right-click (or Control-click) a file or folder and look under Quick Actions or Services for Copy with Copy That. The integration uses a Finder Sync extension plus a macOS Services menu entry visible from any app that supports NSServices.
  4. Linux (GNOME): right-click in Nautilus. The Copy That actions appear via a Nautilus Python extension installed to your user extensions folder.
  5. Linux (KDE): right-click in Dolphin. The actions appear via a KDE ServiceMenu (.desktop file) placed in the KDE services directory during install.
  6. Linux (XFCE / Thunar): right-click in Thunar. The actions appear via a Thunar Custom Action registered at install time.
  7. Selecting any of these actions passes the selected paths to the running Copy That instance (or launches it first if it isn't already open) and immediately opens the drop-staging dialog with those paths pre-filled.
  8. To enable or disable the shell entries, or to opt in to intercepting the default Windows copy handler, go to Settings → Shell.

The global paste hotkey

  1. In your file manager (Explorer, Finder, Files, Nautilus, Dolphin, etc.), copy one or more files normally — Ctrl+C on Windows/Linux or Cmd+C on macOS.
  2. Switch to any other app or stay where you are — you don't need to go back to Copy That.
  3. Press Ctrl + Shift + V (Windows / Linux) or Cmd + Shift + V (macOS) anywhere on your system.
  4. Copy That reads the file paths from the OS clipboard and opens the drop-staging dialog with those sources already filled in.
  5. Choose a destination, confirm the operation, and click Start copying or Start moving.

To change the hotkey combo, go to Settings → General → Global paste shortcut and enter a new combination. To disable the hotkey entirely, toggle it off in the same setting.

The optional Clipboard watcher (Settings → General → Watch clipboard for copied files) polls every 500 ms and shows a small toast notification when file paths appear on the clipboard — handy as a reminder to use the hotkey. Disable it if you prefer a quiet clipboard.

History & totals

  1. Click the History link in the app footer to open the History tab.
  2. Every completed or failed job is listed with columns for: Date, Kind (Copy / Move / Secure delete), Source, Destination, Files transferred, Total size, Status, and Duration.
  3. Filter the list using the controls at the top: filter by date range, job kind, status, or a text search across paths.
  4. To re-run any past job, click the row to select it, then click Re-run — Copy That queues the same source paths and destination immediately, no dialog needed.
  5. To export the filtered list as a CSV, click Export CSV. The file follows RFC 4180 format (double-quoted fields, embedded-quote doubling, newline-safe).
  6. Click the Totals tab (also in the footer) to see lifetime aggregate statistics: total bytes copied, total files, total jobs, and average throughput. A 30-day activity sparkline and a breakdown by job kind are shown below the headline numbers.
  7. To clear all history and statistics, go to Totals → Reset statistics. This is a two-step action — you must confirm twice to prevent accidental deletion.
  8. To limit how long history is retained, go to Settings → Advanced → History retention days and enter a number of days. Leave it blank to keep history forever.

Library — your snapshot timeline

The Library drawer is a unified view of Copy That's content-addressed repository: every copy, sync, and version checkpoint is stored once (deduplicated into content-defined chunks) and surfaced on a single timeline.

  1. Click the Library link in the app footer to open the drawer.
  2. The header shows a dedup “hero” number — how much is physically stored on disk versus the effective (logical) size, and the percentage saved by deduplication.
  3. Switch between four sub-views:
    • Live — the repository at a glance: bytes stored, distinct chunks, snapshot count, plus a Reclaim space button that runs garbage collection to drop chunks no snapshot still references.
    • Snapshots — the unified timeline of every recorded snapshot (copy / sync / version / backup), newest first. Each row offers Restore… (browse the snapshot's files, pick a destination, and restore — with an overwrite / skip / keep-both choice when a file already exists) and Forget (drop the snapshot from the timeline; run Reclaim space to free its bytes).
    • Sources — designate folders for backup, then Back up now to capture an on-demand snapshot (honouring exclude globs). Repeat backups of an unchanged tree are nearly free — identical chunks are stored once.
    • Versions — per-file rolling versions: enter a destination path to list its captured versions.
  4. The existing History, Sync, and Totals drawers remain available — the Library is additive.

Run Copy That as a server

Copy That can run as a headless file server that exposes a folder over the network — via WebDAV, plain HTTP, an S3-compatible API, or SFTP — with optional authentication and a Prometheus /metrics endpoint. You can drive it two ways: the Settings → Server panel in the app, or the copythat serve command for headless machines and scripts.

From the app (Settings → Server)

  1. Click the gear icon in the footer to open Settings, then select the Server tab.
  2. Under Protocols, tick the access methods to expose: WebDAV, HTTP, S3, or SFTP. WebDAV and HTTP can be enabled together; S3 and SFTP each need a server of their own (see the note below).
  3. Set the Bind address. The default 127.0.0.1:8080 serves only to this computer. To reach the server from other machines, use 0.0.0.0:8080 (or your LAN IP) and open that port in your firewall.
  4. Set the Served folder — type a path or click … to pick one. Leaving it blank serves the app's working directory.
  5. Optionally tick Read-only to refuse all uploads and deletions — clients can browse and download, but never change anything on disk.
  6. Choose an Authentication mode — None, Bearer token, or Basic (username + password) — and fill in the credential fields that appear.
  7. Click Start server. The status line shows Running on <address>, and when WebDAV or HTTP is enabled a Metrics URL appears (a Prometheus /metrics endpoint). Click Stop server to shut it down — the server also stops when you quit Copy That.
ProtocolBest forConnect with
WebDAVMounting the folder as a network driveWindows "Map network drive", macOS Finder → Connect to Server (Cmd+K), GNOME Files, or davfs2
HTTPSimple file download / upload (GET / PUT)A web browser, curl, or wget
S3S3-compatible object accessaws CLI, rclone, or any S3 SDK — use Basic auth, where the username / password are the access-key-id / secret (AWS SigV4)
SFTPFile transfer over SSHsftp, scp, FileZilla, WinSCP, or rclone
Protocol combinations: WebDAV and HTTP share one address and can be on at the same time. S3 and SFTP each use their own transport, so enable only one of them at a time on its own server — Copy That refuses to start if you combine S3 or SFTP with another protocol on the same address. S3 also requires Basic or None auth (not Bearer), because it signs requests with AWS SigV4.
Security: a server bound to anything other than 127.0.0.1 with authentication set to None is reachable — read/write unless Read-only — by every device that can reach the address, and Copy That logs a loud warning. For anything beyond your own machine, set Bearer or Basic auth, or put the server behind a reverse proxy that handles authentication.

From the command line

Copy That also ships a command-line binary, copythat, for running a server on a headless machine or from a script. The flags mirror the panel:

  1. Open a terminal and run, for example:
    copythat serve --webdav --root /data --bind 0.0.0.0:8080 --token s3cr3t
  2. With no protocol flag it serves WebDAV; with no --bind it listens on 127.0.0.1:8080.
  3. The terminal prints the bound address and the /metrics URL. Press Ctrl-C to stop the server.

The Settings → Server panel also stores an OpenTelemetry endpoint and webhook destinations (Slack, Discord, ntfy.sh, Pushover) alongside the rest of your settings. OpenTelemetry trace export is activated when you launch the server from the command line with --otel-endpoint.

Migrate from another backup tool

Copy That's repository format is CDR-0 (Common Dedup Repository v0), an open, public-domain specification so any compliant tool can read any compliant repository. The copythat migrate command imports another tool's repository into a CDR-0 repo.

  1. restic, Borg, and Kopia repositories are all supported. Copy That decrypts the source, reconstructs every file, and re-ingests it into a fresh CDR-0 repository — deduplicated with Copy That's own chunking: 
    copythat migrate restic /path/to/restic-repo /path/to/cdr-repo --password <passphrase>
copythat migrate borg   /path/to/borg-repo   /path/to/cdr-repo --password <passphrase>
copythat migrate kopia  /path/to/kopia-repo  /path/to/cdr-repo --password <passphrase>
    (for restic you may also set the RESTIC_PASSWORD environment variable instead of --password).
  2. Detection: point copythat migrate at any repository and it identifies whether it is restic, Borg, Kopia, or CDR-0 from its on-disk layout.
  3. Every importer is validated to restore byte-for-byte against the original files. See the CDR-0 specification for the format details.

Change settings

  1. Click the gear icon in the app footer to open the Settings modal.
  2. Use the tabs to navigate between setting groups:
    • General — language, theme (auto / light / dark), launch at startup, minimize to tray, error prompt style, global paste hotkey combo, clipboard watcher.
    • Transfer — buffer size (64 KiB–16 MiB), verify algorithm, concurrency (1–16 threads or auto), reflink preference, fsync on close, preserve timestamps / permissions / ACLs.
    • Filters — include/exclude glob patterns (e.g., **/*.log), file size bounds, modification-time bounds, skip hidden / system / read-only files.
    • Shell — enable/disable context-menu entries, opt in to intercepting the default Windows copy handler, show job-completion notifications.
    • Secure delete — default shred method, require double confirmation.
    • Advanced — log level, error policy (Ask / Skip / Retry / Abort), history retention, database file path.
    • Updates — auto-check on launch, release channel (stable / beta), skip-this-version.
    • Profiles — save the current settings as a named profile, load a saved profile, or import/export profiles as JSON files for sharing across machines.
  3. All settings take effect immediately — no restart required. Changes hot-reload: if you change the buffer size and then start a new job, that job uses the new value.
  4. Close the Settings modal when done.

Tips & troubleshooting

  1. "Path rejected — contains parent-directory (..) segments"
    A relative path containing .. was passed to Copy That. Retry the operation from your file manager using the shell integration. If you believe the path was valid, please file an issue.
  2. Copy stalls at 100%
    Check the state pill in the header. If it shows Verifying, the post-copy hash pass is running in the background — this is normal, especially for large files or slower algorithms like SHA-256. Wait for it to finish, or disable verify in Settings → Transfer if you don't need it.
  3. Why is my copy slow?
    While a transfer runs, the header shows a small bottleneck badge — a cause icon plus a live throughput sparkline — whenever the speed drops below what the copy should be sustaining. Hover it for the cause: source disk, destination disk, network, antivirus, CPU, or thermal throttling. A saturated destination disk (📤) or a busy antivirus scanner (🛡) are the usual culprits for a local copy.
  4. "Low space on the destination" modal
    Copy That detected that the source files won't fit at the destination. Click Pick which to copy… to open the subset picker, then uncheck items until the size fits the available space.
  5. Windows: "File in use" error
    Copy That opens source files with shared read access and retries automatically with exponential backoff on sharing-violation errors. If the retry loop is exhausted, the holding process is refusing all coexistence — close the application that has the file open and try again.
  6. Linux: AppImage won't launch
    Run chmod +x copythat-*.AppImage to make the file executable. On older glibc systems, use the .deb or .rpm package instead — the AppImage is built against Ubuntu 22.04's glibc.
  7. History tab is empty
    Check Settings → Advanced → Database path. The default locations are:
    • Windows: %LOCALAPPDATA%\CopyThat2026\history.db
    • macOS: ~/Library/Application Support/CopyThat2026/history.db
    • Linux: $XDG_DATA_HOME/CopyThat2026/history.db
    If the path has been changed to a location that doesn't exist, point it back to the correct folder and restart the app.
  8. Getting more help
    Open an issue on GitHub with your OS version, Copy That version, and the text of any error message.

Privacy

  1. No telemetry. Copy That never phones home. The telemetry field in settings is permanently disabled — it cannot be re-enabled by editing the settings file because the field is marked #[serde(skip_deserializing)] in the source code.
  2. No network calls during file operations. While copying, moving, verifying, or securely deleting files, Copy That makes zero network requests.
  3. Update checks are opt-in. The launch update check can be turned off in Settings → Updates. When enabled, it fetches only the update manifest — not your file paths or any usage data — and is throttled to once every 24 hours.
  4. Logs never contain file contents. Debug logs may include file paths and hash values for diagnosing failures, but never include the contents of any file. Hash sidecar exports use job-root-relative paths only.
